# 02. 企画ヒアリングチャット（Phase 2）

Phase 2 では、企画を日本語対話でヒアリングするチャット画面を実装する。第1段階では「自然な対話ができる状態まで」を実装し、第2段階で会話からの構造化データ抽出とサイドパネルでの可視化を追加した。第3段階（最終段階）として、ヒアリング完了後にクラファンページの文章一式（タイトル・キャッチコピー・本文・リターン一覧）をGemini APIで生成し、画面上のプレビューに表示する機能を追加した。DB保存・公開ページとしての表示は引き続き対象外。

## 動作確認

2026-07-02、ローカル環境（`npm run dev`）で以下を確認した。

- 未ログイン状態で `/create` にアクセスすると `/login` へリダイレクトされる（Server Component での判定、`307`）。
- 未ログイン状態で `/api/chat` に直接 POST すると `401 Unauthorized` が返る。
- （認証チェックを一時的に無効化した状態で）実際に Gemini API を呼び出し、ユーザーの回答内容を踏まえた自然な追加質問が返ってくることを確認した。複数ターンにわたり会話の文脈（直前の回答）を踏まえた応答になっている。
- チャットUIは、ユーザーの発言が右寄せ（黒背景）、AIの発言が左寄せ（グレー背景）の吹き出し形式で表示される。
- 実際にユーザーが会話を試した結果を踏まえ、以下の問題を修正・確認済み。
  - `gemini-2.5-flash` のデフォルトの内部思考（thinking）出力が応答に混ざる問題 → `thinkingConfig: { thinkingBudget: 0 }` で解消。
  - AIが未実装機能（ページ生成等）に言及してしまう問題 → システムインストラクションで明示的に禁止。
  - 深掘り質問が「質問攻め」になる問題 → 深掘り回数の上限と、曖昧な回答時の選択肢提示ルールを追加。
  - Gemini API無料枠のクォータ超過（1日20リクエスト、`429 RESOURCE_EXHAUSTED`）でエラーになる問題 → エラーハンドリングを追加し、ユーザー向けに分かりやすいメッセージを表示。根本的にはAPIキーの利用プラン側の制約であり、コード側では解決できない。
- 構造化データ抽出（`responseJsonSchema` によるJSON構造化出力）を実際にGemini APIへ複数ターン投げて検証。当初、モデルに「会話全体を踏まえた最新状態を毎回返す」よう指示していたところ、**会話が進むと以前のターンで抽出済みの項目を省略し、空文字列を返すことがある**（＝以前の内容を後退させてしまう）ことを確認。その後、コードレビューを経て「今回のターンの差分だけを返す」方式（各項目を「新規/更新した内容の文字列」「今回言及なしの空文字列`""`」「明示的な訂正・撤回を表すnull」の3状態で表現）に設計変更し、`""`は前の値を保持・`null`は明示的にクリアという形で、後退バグと訂正・撤回への対応を両立できることを実際のAPI呼び出しで複数ターン検証済み（詳細は「クライアント側のマージ処理」参照）。
- コードレビューで検出された以下の問題も修正済み。
  - 「最初からやり直す」ボタン押下時、送信中の古いリクエストの応答が後から届いてリセット後の状態を上書きしてしまう競合状態 → `AbortController` による中断と、リクエストの世代チェックで解消。
  - AIの応答`message`が空文字列の場合に空のAI発言バブルが表示されてしまう問題 → サーバー側のバリデーションで空文字列も不正な応答として扱うよう修正。
  - `res.json()` の戻り値が無型だった問題 → `ChatApiResponse` 型を定義して明示。
- 実際に会話を試した結果、AIの発言が共感の相槌だけで終わり、質問を含まないままユーザーの入力待ちになってしまう（ユーザーが次に何をすべきか分からなくなる）問題を確認。「質問強制」の仕組み（詳細は下記）を実装し、実際にGemini APIへ「質問を含まない発言」→「質問を強制する追加発言」の2ターンを投げて、追加発言が確実に質問で終わることを検証済み。あわせて、通常時（強制なし）の1回目の発言も、既存のプロンプト設計により大半のケースで自然に質問を含んでいることを確認した。
- 実際の利用中に「AIとの通信に失敗しました。」というエラーが一時的に発生する事象を確認。開発サーバーのログを調査した結果、原因はコードの不具合ではなく **Gemini API側の一時的な過負荷（`503 UNAVAILABLE`, "This model is currently experiencing high demand."）** だった。数秒〜数十秒待って再送信すれば解消する性質のエラーであり、`429`（クォータ超過）と同様、アプリ側では発生自体を防げない外部要因。
  - このエラー調査の過程で、送信失敗時にユーザーの発言だけが会話履歴に残ってしまい、再送信すると同じ発言が重複して表示される不具合を発見。送信は「ユーザー発言を履歴に追加 → API呼び出し」という順序で行っており、API呼び出しが失敗しても追加済みの発言はそのまま残っていたため。失敗時に送信前の会話履歴・入力欄の状態にロールバックするよう修正し、重複が起きないようにした（`ChatPanel.tsx`）。
- ページ生成機能（`/api/generate-page`）について、認証チェックを一時的に無効化した状態で以下を確認した。
  - `extracted` を渡さない、または4項目のいずれかが空のリクエストは `400` になる（サーバー側でも未完了状態からの生成をブロックする、詳細は後述）。
  - 4項目が揃った実際のヒアリング内容を渡し、Gemini APIを実際に呼び出して `title` / `tagline` / `story` / `rewards` を含むJSONが返ってくることを確認した。ヒアリング内容に明記されていない金額を数値として創作しないことも確認した（後述のプロンプトの「厳守事項」で明示的に指示）。
  - `npm run build` でのビルド（Turbopack, Next.js 16.2.9）が成功し、`/api/generate-page` がRoute Handlerとして認識されることを確認した。
- ブラウザでの実操作（ボタン押下・タブ切り替え等）は、マジックリンク認証の性質上、開発時点では自動検証していない（下記と同様、ユーザー自身のブラウザでの動作確認を別途実施する）。

マジックリンク経由の実ブラウザでのログイン → `/create` での一連の操作は、認証フローの性質上（メール受信が必要）自動検証はしていない。ユーザー自身のブラウザでの動作確認を都度実施している。

## 画面構成

| パス | 種別 | 内容 |
| --- | --- | --- |
| `/create` | Server Component（`src/app/create/page.tsx`） | `supabase.auth.getUser()` でログイン判定し、未ログインなら `/login` へリダイレクト。ログイン済みなら `ChatPanel` を描画する。 |
| `src/app/create/ChatPanel.tsx` | Client Component | チャットUI本体。メッセージ一覧・入力欄・送信ボタン・リセットボタンを持ち、`ExtractedPanel` を並べて表示する。ヒアリング完了後は「チャット」「プレビュー」タブで `PagePreview` と切り替え表示する。 |
| `src/app/create/ExtractedPanel.tsx` | Client Component（Presentational） | 抽出済み4項目をカード状に表示するサイドパネル。ヒアリング完了時は「ページを生成する」ボタンも表示する。 |
| `src/app/create/PagePreview.tsx` | Client Component（Presentational） | ページ生成結果（タイトル・キャッチコピー・本文・リターン一覧）を表示するプレビュー画面。 |
| `src/app/api/chat/route.ts` | Route Handler（POST） | ログイン判定後、会話履歴を Gemini API に渡し、会話文と抽出済み構造化データの両方を返す。 |
| `src/app/api/generate-page/route.ts` | Route Handler（POST） | ログイン判定後、確定済みの4項目（`ExtractedFields`）を Gemini API に渡し、ページ生成結果（`GeneratedPageContent`）を返す。 |

### チャットUIの挙動

- 画面表示時、AIからの最初の質問（企画の目的を尋ねる固定メッセージ）を表示する。Gemini APIは呼ばない。
- ユーザーがメッセージを送信すると、それまでの会話履歴全体（`ChatMessage[]`）を `/api/chat` に POST する。
- 応答が返るまでは「考え中...」を表示し、入力・送信を無効化する。
- 通信エラー時は吹き出しの下にエラーメッセージを表示する（429の場合は専用の文言）。
- 「最初からやり直す」ボタンで、会話・抽出済みデータ・入力欄・エラー表示・生成済みページ・プレビュー表示状態をすべて初期状態に戻せる。送信中のリクエストがあれば `AbortController` で中断し、中断後に届いた古い応答は無視する（リセット後の状態を上書きしない）。
- 会話・抽出データ・生成済みページはブラウザの状態（React state）にのみ保持し、DB等への永続化は今回も行わない。ページをリロードすれば消える。

### サイドパネル（`ExtractedPanel`）の挙動

- デスクトップ幅（Tailwindの `md` 以上）ではチャット欄の右側、モバイル幅ではチャット欄の下部に表示する（`flex flex-col md:flex-row` で実現。DOM順はチャット→パネルの順で、モバイルでは自然に下に積まれる）。
- 「目的」「ターゲット」「ストーリー」「リターン設計」の4項目をそれぞれカードで表示し、値が空の項目は「未入力」とグレー表示する。
- AIの応答が返るたびに、抽出済みデータをその場で更新する（ポーリングやリアルタイム通信は使わず、チャット応答と同じレスポンスに含めて返す）。
- 4項目すべてが埋まったら、パネル上部に「ヒアリング完了」の表示と「ページを生成する」ボタンを出す（詳細は次節「ページ生成」を参照）。

## ページ生成

ヒアリング完了後、これまでの4項目（目的・ターゲット・ストーリー・リターン設計）をもとに、クラファンページの文章一式をGemini APIで生成し、画面上に表示する機能。DB保存・公開ページとしての表示は対象外で、生成してその場で確認できるところまでを実装した。

### 生成される内容

`GeneratedPageContent`（`src/types/chat.ts`）として以下を生成する。

- `title`: プロジェクトの見出しとなるタイトル。
- `tagline`: 「一言で言うと何のプロジェクトか」が伝わるキャッチコピー・要約。
- `story`: 目的・背景・想いを、支援者に伝わるように構成した本文（複数段落のプレーンテキスト）。
- `rewards`: リターン一覧。ヒアリングで聞いた内容を支援者向けに整理したもので、各要素は `title`（リターン名）と `description`（説明文）を持つ配列（`PageReward[]`）。

### UI（ボタン・プレビュー表示）

- `ExtractedPanel` の「ヒアリング完了」表示の下に「ページを生成する」ボタンを表示する（4項目がすべて埋まっている場合のみ）。生成中は「生成中...」表示でボタンを無効化し、二重送信を防ぐ。生成済みの状態で再度押すと「ページを再生成する」という表示になり、何度でも生成し直せる（直前の生成結果を上書きする）。
- 生成に成功すると、`ChatPanel` は自動的に「プレビュー」タブへ切り替わり、`PagePreview` を表示する。生成済みページがある場合のみ、チャット画面の見出し横に「チャット」「プレビュー」を切り替えるタブを表示し、ヒアリングを続けながら任意のタイミングでプレビューを見返せるようにしている。
- 「チャット画面とは別の見やすいプレビュー画面」という要件に対し、サイドパネル（`md:w-72`という限られた幅）に埋め込むのではなく、タブ切り替えで全幅を使った独立したプレビュー画面として表示する構成にした。ページ遷移は行わず、同じ `/create` ページ内で完結する。
- 生成に失敗した場合は、`ExtractedPanel` 内のボタン下にエラーメッセージを表示する（429の場合はチャットと同様の専用文言）。プレビュー自体への遷移は行わない。
- 「最初からやり直す」を押すと、生成済みページ・プレビュー表示状態もすべてリセットされる。

### AIへのプロンプト設計（ページ生成用）

`src/app/api/generate-page/route.ts` 内の `SYSTEM_INSTRUCTION` として実装。チャット用の `SYSTEM_INSTRUCTION`（`/api/chat/route.ts`）とは役割が異なる（対話ではなく1回限りの文章生成）ため、別のプロンプトとして分離している。

設計方針：

- 役割を「クラウドファンディングページの文章作成を手伝う、プロのライター兼編集者」と定義し、ヒアリングで得た4項目（purpose/target/story/reward）を渡して、`title` / `tagline` / `story` / `rewards` を生成させる。
- `story` は複数段落のプレーンテキストとし、Markdown記法（見出し記号や箇条書き記号など）を使わないよう明示する。フロント側の表示が単純な `whitespace-pre-wrap` によるプレーンテキスト表示のため、Markdown記法が混ざると記号がそのまま表示されてしまうことを避ける狙い。
- 「厳守事項」として、ヒアリング内容の範囲を超えた誇張をしないこと、特に金額・個数・期間などの具体的な数字はヒアリング内容に明記されている場合のみ使用し、明記されていない数字を創作しないことを明示している。ヒアリングでは金額等が曖昧なまま進むケースがあり得るため、モデルが数字を捏造して支援者に誤った情報を伝えてしまうリスクを避けるための指示。実際に金額が明記されたヒアリング内容を渡して検証し、明記された金額のみを使い、それ以外の数字を創作しないことを確認済み（詳細は「動作確認」参照）。
- チャット用プロンプトと同様、常に日本語・出力はJSON形式のみとする指示を含める。

### 技術構成（ページ生成）

- チャットと同じ `@google/genai` クライアント・同じモデル（`gemini-2.5-flash`）を使用する。ページ生成は「ヒアリング完了後に1回ユーザーが能動的に押すボタン」による単発の呼び出しであり、チャットのように1会話で何度も呼ばれるものではないため、無料枠クォータへの影響はチャットに比べて小さい。モデルを変える運用上のメリットが今のところ薄いため、シンプルさを優先して同じモデルを使う判断とした。
- `responseMimeType: "application/json"` と `responseJsonSchema` による構造化出力を使用し、`{ title, tagline, story, rewards: { title, description }[] }` の形にスキーマ強制する（`/api/chat` と同じ方式）。`thinkingConfig: { thinkingBudget: 0 }` も同様に指定する。
- サーバー側で `response.text` を `JSON.parse` し、`title` / `tagline` / `story` が非空文字列であること、`rewards` が `{ title: string; description: string }` の配列であることを検証してから返す。不正な形式の場合は `InvalidModelResponseError` を投げ、`502`（`invalid_model_response`）を返す（`/api/chat` と同じエラー分類方針）。
- リクエストボディの `extracted`（`ExtractedFields`）について、4項目すべてが埋まっていること（`isExtractedComplete`）をサーバー側でも検証し、未完了の場合は `400` を返す。クライアント側のボタンは4項目が揃っている場合のみ表示されるが、`/api/generate-page` を直接叩く他のクライアントがあっても未完了状態からの生成を防げるようにするため。
- 認証チェック（`supabase.auth.getUser()`、未ログインなら `401`）は `/api/chat` と同じ方針。

### 共通化した「ヒアリング完了」判定

4項目すべてに値が入っているかどうかの判定（`isExtractedComplete`）を `src/types/chat.ts` の共有関数として切り出し、`/api/chat/route.ts`（質問強制の要否判定）・`ExtractedPanel.tsx`（「ヒアリング完了」表示）・`ChatPanel.tsx`（タブ表示や生成ボタン活性化の判断に間接的に利用）で共有する。今回ページ生成ボタンという3つ目の利用箇所が増えるにあたり、これ以上ロジックが個別に重複するのを避けるため、既存の2箇所（`route.ts`・`ExtractedPanel.tsx`で個別に実装されていた同一ロジック）も含めて共通化した。

## AIへのプロンプト設計

`src/app/api/chat/route.ts` 内の `SYSTEM_INSTRUCTION` として実装。ヒアリング対象は以下の4項目（内部キー名も併記）：

1. purpose（目的）: 企画の目的（何を実現したいか）
2. target（ターゲット）: 誰に向けたプロジェクトか
3. story（ストーリー）: 背景・想い
4. reward（リターン設計）: 支援者に何を返すか

設計方針：

- 一度に1つだけ質問し、フォームの穴埋めのような機械的な質問にしない。
- 直前の回答を踏まえて自然に次の質問へつなげる（4項目の順序には固執しない）。
- 同じ項目の深掘りは最大1〜2回まで。曖昧な回答（「分からない」等）には深掘りを重ねず、具体例・選択肢を2〜3個提示して選ばせるか、次に進む提案をする。ユーザーにプレッシャーを与えない、優しく寄り添うトーンを保つ。
- 4項目をひととおり聞けたら要約し、感謝を伝える。未実装機能（ページ生成等）には一切言及しない。
- 常に日本語、丁寧だが堅すぎない温かみのある口調。
- 会話に専念し、企画書やページ文章そのものは生成しない（Phase 2のスコープ外）。
- 出力形式として、`message`（会話文）と `extracted`（4項目の抽出結果）を含むJSONを毎回返すことを指示。`extracted` の各項目は「今回のターンの差分」のみを表現させ、次の3状態のいずれかを返すよう指示する。
  1. 今回新しく分かった／更新された内容 → その文字列
  2. 今回言及がなかった → 空文字列 `""`（前のターンまでの内容はアプリ側が保持するため、モデルが繰り返す必要はない）
  3. ユーザーが明示的に訂正・撤回した → `null`（その項目をアプリ側で未入力に戻す合図）

### 質問強制の仕組み（無反応対策）

AIの発言が共感の相槌だけで終わり、質問を含まないままだと、ユーザーは次に何をすればよいか分からなくなる（自分から話しかけないと会話が進まない）。これを防ぐため、`route.ts` に以下の仕組みを実装している。

1. 通常どおり1回目の発言（`message`）を生成する。1回目の生成では、これまでどおり自然な会話を許容する（相槌・共感だけで終わってもよい）。
2. 生成された `message` の末尾が「？」「?」または「〜か」で終わっているかを簡易判定する（`endsWithQuestion()`）。
3. 質問を含まないと判定され、かつヒアリングがまだ完了していない（4項目のいずれかが未入力）場合のみ、ユーザーの入力を待たずにもう一度だけ追加の発言を生成する。この2回目の生成では、通常のシステムインストラクションに加えて「必ず質問で終わらせること」を絶対条件とする追加指示（`FORCE_QUESTION_INSTRUCTION`）を付与し、1回目の発言を会話履歴に含めた上で呼び出す。
4. 1回目と2回目の発言を`\n\n`で連結し、1つのAI発言としてユーザーに表示する（クライアント側の変更は不要。既存の1つの吹き出しの中で改行される）。
5. 追加生成は最大1回まで。2回目の生成でも質問が含まれていなかった場合、それ以上は繰り返さずそのまま表示する。2回目の生成自体が失敗した場合（レート制限・不正な応答など）も、1回目の発言のみを返してユーザー体験を壊さないようにする。
6. 4項目すべてが埋まっている（ヒアリング完了）場合は、この仕組みを適用しない。完了時の要約・感謝の発言は質問で終わる必要がないため。

ヒアリング完了の判定には、クライアントから送信される「直前までの確定済み抽出データ」（`previousExtracted`、後述）と1回目の生成結果をマージした後の状態を使う。

## 技術構成

- Gemini API 呼び出しに `@google/genai`（Google Gen AI SDK for TypeScript/JavaScript）を使用。
  - `GEMINI_API_KEY`（`.env.local`、サーバー専用・`NEXT_PUBLIC_` プレフィックスなし）を使用してクライアントを生成。
  - モデルは `gemini-2.5-flash` を使用（会話用途のため軽量・高速なモデルを選定）。
  - `ai.models.generateContent()` に会話履歴全体（`role: "user" | "model"` の配列）と `config.systemInstruction` を渡し、都度リクエストとして呼び出すステートレスな方式（サーバー側でセッションを保持しない）。
  - `thinkingConfig: { thinkingBudget: 0 }` で内部思考出力を無効化。
- **構造化出力**: `config.responseMimeType: "application/json"` と `config.responseJsonSchema` を指定し、`{ message: string, extracted: { purpose, target, story, reward } }` の形にスキーマ強制した1回のAPI呼び出しで、会話文と構造化データの両方を取得する。
  - 会話文と抽出処理を別々の2回のAPI呼び出しに分ける方式（Option B）ではなく、1回の呼び出しで両方を得る方式（Option A）を採用。理由は、Gemini APIの無料枠クォータが1日20リクエストと非常に少なく、2倍のリクエスト数になる方式は現実的でないため。
  - `@google/genai` の構造化出力には `responseSchema`（`Type` enumを使う独自形式）と `responseJsonSchema`（標準のJSON Schema）の2種類があり、依存を増やさずシンプルに書ける後者を採用。
  - `ai.models.generateContent()` のレスポンスはJSON文字列として返るため、`route.ts` 側で `JSON.parse` し、各フィールドの型を検証してから返す（不正な形式の場合は `502` を返す）。
- `/api/chat` は Route Handler として実装し、`src/lib/supabase/server.ts` の `createClient()` で `supabase.auth.getUser()` を呼び、未ログインなら `401` を返す（[`docs/01_AUTH.md`](01_AUTH.md) のログイン判定方針に準拠）。Gemini APIのレート制限（`429`）は専用のレスポンスとして返す。
- `/create` の認証チェックは Server Component（`page.tsx`）内で行い、未ログインなら `redirect("/login")`。
- 型定義（`src/types/chat.ts`）
  - `ChatMessage`（`role: "user" | "model"`, `content: string`）: `ChatPanel.tsx` と `route.ts` で共有する会話履歴の型。
  - `ExtractedFields`（`purpose` / `target` / `story` / `reward`, いずれも `string`）: 画面に表示する、確定済みの抽出データの型。
  - `ExtractedFieldUpdates`（各項目 `string | null`）: Geminiが1回の生成ごとに返す「差分」の型。`""`＝今回言及なし、`null`＝明示的な訂正・撤回、それ以外の文字列＝新規/更新内容。
  - `EMPTY_EXTRACTED_FIELDS`: `ExtractedFields` の初期値・リセット用定数（全項目が空文字列）。
  - `ChatApiResponse`（`{ reply: string; extracted: ExtractedFields }`）: `/api/chat` の成功時レスポンスの型。`extracted` はサーバー側でマージ済みの確定状態（差分ではない）。`ChatPanel.tsx` 側で `res.json()` の戻り値をこの型として扱い、`setExtracted(data.extracted)` でそのまま置き換える。
  - `mergeExtractedFields(prev, updates)`: `ExtractedFieldUpdates` を `ExtractedFields` にマージする純粋関数。`null` は該当項目を空文字列にクリアし、`""` は前の値を維持し、それ以外の文字列は新しい値で上書きする。これにより「省略による後退を防ぐ」ことと「ユーザーの訂正・撤回を反映できる」ことを両立している。
  - `isExtractedComplete(extracted)`: 4項目すべてに値が入っているか（ヒアリング完了か）を判定する純粋関数。`/api/chat/route.ts`（質問強制の要否判定）・`/api/generate-page/route.ts`（未完了状態からの生成をブロックするサーバー側バリデーション）・`ExtractedPanel.tsx`（「ヒアリング完了」表示）で共有する。
  - `PageReward`（`{ title: string; description: string }`）・`GeneratedPageContent`（`{ title, tagline, story, rewards: PageReward[] }`）: ページ生成結果の型。
  - `GeneratePageApiResponse`（`{ page: GeneratedPageContent }`）: `/api/generate-page` の成功時レスポンスの型。
- サーバー側のマージ処理（`route.ts`）: クライアントはリクエストボディに `previousExtracted`（直前までの確定済み `ExtractedFields`）を含めて送る。サーバーは1回目（および発生した場合は2回目）の生成結果を `mergeExtractedFields` で `previousExtracted` に順にマージし、確定済みの `ExtractedFields` としてレスポンスに含める。マージ処理をサーバー側に一本化したことで、`/api/chat` を呼ぶ他のクライアントがあっても同じ「後退防止・訂正反映」の挙動が保証される（以前はクライアント側だけで行っていた）。

### コストへの影響（Gemini API呼び出し回数）

質問強制の仕組みにより、1回のユーザー発言に対してGemini APIを最大2回呼び出す可能性がある（通常時は1回のまま）。実際に確認した限り、既存のプロンプト設計（「一度に1つだけ質問する」という指示）により、1回目の生成は多くの場合すでに自然に質問で終わっており、2回目の追加生成が発生するのは「AIが質問を含めずに応答を終えてしまった」場合に限られる。また、ヒアリングが完了した後は仕組み自体が発動しない。そのため、呼び出し回数の増加は最悪でも2倍程度で、実際の増加率はそれより小さく、発生頻度はモデルの応答傾向に依存する（正確な発生率は本番運用のログを見ないと分からない）。Gemini APIの無料枠クォータ（1日20リクエスト）は元々余裕がない水準のため、この機能追加によりクォータ超過（`429`）が従来よりも早く発生しやすくなる点には注意が必要。

## 今後の拡張予定（オープンな論点）

- 生成したページ文章・リターン一覧の永続化（DB保存）、および公開ページとしての表示・編集は未実装。今回はGemini APIで生成してプレビュー表示するところまで。
- 会話履歴・抽出データ・生成済みページの永続化（DB保存、途中離脱からの再開）は未実装。今回はブラウザのState内のみ。
- Gemini APIのレスポンス遅延・エラー時のリトライ方針は未検討（チャット・ページ生成のいずれも）。
- Gemini APIの無料枠クォータ（1日20リクエスト）は開発・検証だけでもすぐ枯渇する水準のため、継続利用する場合はAPIキーの課金プラン移行を検討する。ページ生成の追加によりチャットと合わせた呼び出し回数がさらに増える。
- `ExtractedFields` の4項目名（purpose/target/story/reward）が `ExtractedPanel.tsx`・`route.ts`・`ChatPanel.tsx`・`generate-page/route.ts` の4箇所に個別にハードコードされており、単一の情報源がない（詳細は `CLAUDE.md` の「既知の改善候補」を参照）。「ヒアリング完了」の判定ロジック自体は `isExtractedComplete` として共通化済みだが、フィールド名の列挙は依然として複数箇所に残っている。
- Gemini APIが構造化出力のスキーマに反する不正な形式を返した場合のエラー分類が粗い（詳細は `CLAUDE.md` の「既知の改善候補」を参照）。ページ生成（`/api/generate-page`）でも同様の課題がある。
- ページ生成結果（特に `story` や `rewards` の内容）にユーザーが手を加えて編集する機能はまだない。生成結果をそのまま確認するのみ。
