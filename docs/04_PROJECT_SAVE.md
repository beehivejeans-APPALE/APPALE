# 04. プロジェクトの自動保存・復元

[`docs/03_DATABASE.md`](03_DATABASE.md) で設計した `projects` テーブルを使って、`/create` でのヒアリング内容・会話ログ・生成済みページ内容を自動的に保存し、再訪問時に復元する機能を実装した。ステータス（下書き/レビュー待ち/公開済み）の切り替えUIと、公開ページとしての表示は対象外（引き続き未実装）。

## スコープ・前提

- ユーザーごとに「進行中のプロジェクトは1件」という前提を置いている。プロジェクトの一覧表示・複数プロジェクトの新規作成UIはまだないため、`/create` を開くと常にそのユーザーの最新のプロジェクト行を読み書きする。将来複数プロジェクトを持てるようにする場合、この前提（`/api/project` が暗黙的にユーザーの最新行を対象にする実装）ごと見直しが必要になる。
- 会話・抽出データ・生成済みページの3種類すべてを同じ1行（`projects` テーブルの該当ユーザーの行）に保存する。[`docs/02_CREATE_CHAT.md`](02_CREATE_CHAT.md) 時点では「ブラウザのStateにのみ保持し、リロードすれば消える」という制約があったが、本実装でその制約を解消した。

## 画面構成の変更

| パス | 変更内容 |
| --- | --- |
| `src/app/api/project/route.ts` | 新規追加。`POST`: ユーザーの既存プロジェクトを返すか、なければ新規作成して返す（get-or-create）。`PATCH`: 会話ログ・抽出データ・生成済みページ内容を部分更新する。 |
| `src/types/project.ts` | 新規追加。`projects` テーブルの1行に対応する `ProjectRow` 型、`ChatPanel` が扱いやすい形に整えた `ProjectState` 型、両者を変換する `rowToProjectState()` を定義。 |
| `src/app/create/ChatPanel.tsx` | 初回マウント時に `/api/project`（`POST`）を呼び、既存プロジェクトの復元 or 新規作成を行うように変更。会話・抽出データの変更をデバウンス保存、ページ生成結果を即時保存する仕組みと、保存状態インジケーターを追加。 |

## 復元の仕組み

- `ChatPanel` は表示時に `useEffect` で `/api/project` に `POST` する。
  - サーバー側（`route.ts`）は、ログイン中のユーザーの `projects` 行を `user_id` で検索し、存在すればそれを返し、存在しなければ空の行を1件 `insert` して返す（`purpose`/`target`/`story`/`reward` は空文字列、`messages` は空配列がテーブルのデフォルト値のため、特別な初期化コードは不要）。
- レスポンスの `messages` が空配列の場合（＝新規プロジェクト）は、従来どおり固定の初回メッセージ（`INITIAL_MESSAGE`）を表示する。空でない場合はそれをそのまま会話履歴として復元する。
- 抽出データ（`extracted`）・生成済みページ（`generatedPage`）も同じレスポンスに含まれ、そのまま画面の状態にセットする。
- 読み込みが完了するまでは「読み込み中...」を表示し、チャットUI自体を描画しない（復元前の固定初回メッセージが一瞬表示されてから復元後の内容に置き換わる、というちらつきを避けるため）。
- `/api/project` の呼び出しに失敗した場合は、エラーメッセージを表示してチャットUI自体を描画しない（再読み込みを促す）。

## 自動保存の仕組み

保存タイミングを2種類に分けている。

1. **デバウンス保存**（`scheduleSave`）: チャットの送信が成功し、会話ログ（`messages`）と抽出データ（`extracted`）が更新されたタイミングで呼ぶ。呼び出しのたびに保留中のタイマーをリセットし、1.5秒（`AUTOSAVE_DEBOUNCE_MS`）変更が止まってから実際に `PATCH` する。連続してすぐに次のやり取りが発生した場合に、保存リクエストが都度発生するのを避ける狙い。
2. **即時保存**（`saveImmediately`）: 以下の、ユーザーの明示的な操作の直後に、保留中のデバウンスタイマーを打ち消してすぐに `PATCH` する。
   - ページ生成成功時: 生成された `title`/`tagline`/`story`/`rewards` を保存する。
   - 「最初からやり直す」押下時: `messages` を空配列、`extracted` を全項目空文字列、`generatedPage` を `null` にして保存する。これにより、リセット後に再度ページを開いても、リセット前の内容が復元されてしまうことを防ぐ（自動保存機能を追加したことで新たに気をつける必要が生じた点。保存を行わずにローカルStateだけリセットすると、リロード時に古い内容が復元されてしまい「やり直す」ボタンが実質的に機能しなくなるため）。

いずれも `/api/project` の `PATCH` を呼ぶ。リクエストボディは渡されたキー（`messages`/`extracted`/`generatedPage`）だけを送り、サーバー側もキーが存在する項目だけを更新する部分更新方式にしている（例: デバウンス保存では `generatedPage` を送らないため、既存の生成済みページ内容は変更されない）。

## 保存状態インジケーター

`ChatPanel` の見出し横に、小さなテキストで保存状態を表示する（`SaveStatusIndicator`）。

- `saving`: 「保存中...」
- `saved`: 「保存済み」
- `error`: 「保存に失敗しました」（赤字）
- 初期状態（読み込み中）は何も表示しない

保存が成功した直後だけでなく、初回読み込み完了時にも「保存済み」を表示する（読み込んだ内容＝DB上の内容と一致している状態のため）。

## `/api/project` の設計

- 認証チェック（`supabase.auth.getUser()`、未ログインなら `401`）は `/api/chat`・`/api/generate-page` と同じ方針。
- `POST`（get-or-create）: `user_id` で最新の1行を検索し、なければ `insert`。特定のプロジェクトIDを指定する引数は取らない（前提どおり、常にそのユーザーの1件を対象にするため）。
- `PATCH`: リクエストボディの `id` で対象行を指定し、`user_id` との一致もあわせて `.eq()` で絞り込む（RLSによって他人の行は更新できないが、複数行を誤って更新しないよう明示的に絞り込んでいる）。`messages`/`extracted`/`generatedPage` はそれぞれ簡単な形状チェック（型・必須プロパティの存在）を行い、不正な形式は `400` を返す。
- レスポンス・保存先マッピングの詳細は `src/types/project.ts` の `ProjectRow` / `ProjectState` / `rowToProjectState()` を参照。

## 今後の拡張予定（オープンな論点）

- 複数プロジェクトの一覧・新規作成・切り替えUIは未実装。「進行中のプロジェクトは1件」という前提を置いているため、実装する際は `/api/project` の設計（特にユーザーの最新行を暗黙的に対象にしている部分）ごと見直しが必要。
- `status` の切り替えUI（下書き→レビュー待ち→公開済み）は未実装。
- 保存失敗時（`error` 状態）に自動リトライする仕組みはない。ユーザーが次に何か操作すれば再度保存が試みられるのみ。
- 同一ユーザーが複数タブで同時に `/create` を開いた場合の競合（片方の変更がもう片方の保存で上書きされる、初回アクセスが同時に発生した場合に行が2件作成される可能性がある等）は未検証・未対応。
